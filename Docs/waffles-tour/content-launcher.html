8 · Content launcher

The launcher is the on-screen menu: a panel that browses content (folders, media, playlists), drills into folders, searches, and opens whatever the user taps. The UI is bridge-agnostic; a separate adapter turns your CMS/bridge schema into a generic item list. Swap the adapter and the same launcher drives any data source.

Key files

  • Bridge/qml/DsContentLauncherModel.qml — the bridge adapter (schema → items)
  • waffles/menu_launcher/DsContentLauncher.qml — the launcher panel UI (a DsViewer)
  • ux/DsVirtualKeyboard.qml — docked Qt Virtual Keyboard (search)
  • ux/DsFloatingKeyboard.qml — floating keyboard (web viewer)
UI vs adapter

DsContentLauncher knows nothing about the bridge — it consumes a model that exposes displayedItems, currentPlaylist, path/atRoot, enter()/back(), and the search properties. DsContentLauncherModel is the supplied adapter for the Downstream bridge; write a different model with the same surface to drive the launcher from anything.

The adapter — DsContentLauncherModel

The adapter reads the running bridge (the platform record plus the active interactive event), classifies each linked record into a kind, and emits a flat item list. Everything is configured by appKey (the schema field names), so it matches a schema without code changes.

Per-type overrides let one model serve several event/platform types: eventFieldOverrides / platformFieldOverrides are { "Type Name": { contentField, playlistField } } maps that win over the globals. interactiveEventTypes (default ["Connect Wall Event","Explore Room Event"]) lists, in priority order, the event type names that carry menu content — the first whose schedule has a current record wins.

Kinds — classify, then refine

Each record is reduced to a kind the UI understands. Type is matched first by the stable type_uid (typeKindByUid), then by type_name (typeKindByName); a record carrying a media descriptor is then refined to image/video/ pdf/weblink from its .type. Resulting kinds: media · image · video · pdf · weblink · folder · playlist · unknown.

Drill vs open

Two distinct facts per item:

propertymeaning
recordHasChildrendata fact — the record actually carries children
hasChildrenUX choice — is this item's kind in drillableKinds?

drillableKinds (default ["folder","playlist"]) decides what a tap does:

  • Drillable kind → tap calls model.enter(item), navigating into its children (the breadcrumb grows).
  • Otherwise → tap fires openRequested(item) for the app to open — even if the record has children. (E.g. drop "playlist" from drillableKinds so a playlist opens & plays instead of drilling into its slides.)

Item descriptor

{
  uid,                // record id
  title,              // from titleFields, else "(untitled)"
  kind,               // media|image|video|pdf|weblink|folder|playlist|unknown
  thumbnail,          // url — media.filepath, else a thumbnailFields image
  media,              // the media descriptor { type, filepath, … } or null
  hasChildren,        // UX: kind ∈ drillableKinds → drill on tap
  recordHasChildren,  // data: record really has children
  record              // the raw bridge record (template-specific fields)
}

Link fields & the comma-split quirk

A multi-value LINK arrives as ["uid1,uid2"]

The bridge serialises a multi-value LINK field as a one-element array holding a comma-joined string — not an array of uids. _uidsFromField(raw) normalises every form (single uid, comma string, or array) by splitting each entry on commas and flattening, so ["uid1,uid2"] → ["uid1","uid2"]. (Splitting a single uid with no comma yields [uid], so single links are unaffected.) Skipping this is why an "additional content" field with several items used to resolve to nothing.

Lifecycle & debugging

The adapter rebuilds itself from bridge signals — it listens (via a typed Connections) to DsBridge.contentChanged(), bridgeUpdated(), and databaseChanged(), each calling rebuild() (which recomputes library/playlists and replays the breadcrumb by uid so a drilled view survives content updates). Methods: enter() / back() / rebuild() / slidesFor(record) (see the playlists page) / dump() & dumpAllChildren() (console snapshots of the resolved tree).

The launcher UI — DsContentLauncher

A DsViewer (viewerType: Launcher) panel, sized in chrome units (viewerWidth: dp(453), viewerHeight: dp(762)) so it scales with uiScale. It takes the model (the adapter) and the hosting stage, draws its own glass (a touch stronger than media chrome — glassTintOpacity: 0.9), and emits one signal:

DsContentLauncher {
    model: launcherModel       // the adapter
    stage: stage              // the DsWaffleStage host
    shown: false             // start hidden; the stage FAB / double-tap re-shows it
    onOpenRequested: (item) => { /* map item → a viewer / playlist, below */ }
}

It has two panes, toggled by tab pills (viewMode = "content" | "search"):

  • Content — a breadcrumb (shown when !model.atRoot, with a back button), the current playlist row (at root, if any), then the library list. Leaf rows emit openRequested; drillable rows call model.enter.
  • Search — type-filter chips, a results list, and a docked keyboard (below).

Search

Search lives on the adapter and the pane just binds to it:

propertywhat it does
searchQueryfree text; case-insensitive substring match on item titles
searchKindskind filter (empty = all); set by the chips (All / Images / Video / PDF / Web / Folders)
searchResultsflat matches (same item shape) by walking the loaded tree; updates reactively

The search field sits at the bottom of the pane (above the keyboard) and grabs focus when the pane opens (raising the keyboard). The clear (✕) button shows when there's text and resets both the field and model.searchQuery. With an empty query and no kind filter, results are empty.

On-screen keyboard

Two keyboards, both wrapping Qt's Virtual Keyboard; only one InputPanel may be active per window (they share one input context), so the launcher's docked panel and the floating one gate each other off.

Docked — DsVirtualKeyboard (search)

Lives inside the launcher's search pane (in a Loader gated to search mode). It exposes panelActive and panelHeight (aliases of the InputPanel); the launcher animates the keyboard area open to panelHeight and disables panel-drag while it's up.

Floating — DsFloatingKeyboard (web viewer)

A draggable framed keyboard (drag bar + title + close + the same DsVirtualKeyboard), hosted on the stage's top layer (modal3) so it floats over everything. Toggle it with DsWaffleStage.floatingKeyboardShown; the web controls' keyboard button flips it, and it's gated off while the launcher's docked keyboard is active. Width is dp(437) (uiScale-aware); it emits closeRequested().

Style & locales (settings)

[waffles.keyboard]
style   = "waffles"                              # VirtualKeyboardSettings.styleName
locales = [["en_US", "es_ES", "fr_FR", "de_DE", "it_IT"]]  # active layouts; globe key cycles

Both are applied at runtime in Component.onCompleted via VirtualKeyboardSettings.styleName / .activeLocales, overriding the QT_VIRTUALKEYBOARD_STYLE bootstrap env var (set in main.cpp alongside QT_IM_MODULE=qtvirtualkeyboard) — so swapping the style/locales needs no recompile. CJK / Arabic locales work too ("zh_CN", "ja_JP", "ko_KR", "ar_AR").

Bundling the waffles style in a static lib

The style ships in the Dsqt.Core module under qrc:/qt/qml/Dsqt/Core/res/QtQuick/VirtualKeyboard/Styles/waffles/, and DsQmlApplicationEngine adds qrc:/qt/qml/Dsqt/Core/res to the QML import path so the keyboard resolves it. The files are listed individually in RESOURCES (not via a .qrc): Core is a static library, and a .qrc's initialiser gets dead-stripped from a static lib, so the style would silently fail to load.

Wiring it up

Give the launcher an adapter and translate openRequested into the right action — open a media viewer, or open & play a playlist:

DsContentLauncherModel { id: launcherModel; drillableKinds: ["folder"] }  // playlists open, not drill

DsContentLauncher {
    model: launcherModel
    stage: stage
    onOpenRequested: (item) => {
        if (item.kind === "playlist")
            stage.openPlaylist({ "slides": launcherModel.slidesFor(item.record),
                                "ambient": false });
        else
            stage.openViewer({ "model": { "title": item.title, "media": item.media },
                              "stage": stage });
    }
}

The mapping stays in the app (not Waffles), so the launcher and viewers stay bridge-agnostic. See the recipes for the full viewer-props mapping, and playlists for what happens after openPlaylist.
