5 · Glass & theming

The frosted-glass effect — how a viewer's chrome shows a blurred copy of everything behind it — plus the two-tier design tokens (DsTheme) and the settings that feed them.

Key files

  • waffles/DsGlassBackground.qml — the reusable glass element
  • waffles/DsWaffleStage.qml — builds & maintains the slot chain
  • waffles/DsTheme.qml — the design-token singleton
  • Docs/articles/color_system.md — the colour philosophy in depth
  • Examples/AssetViewer/settings/waffles_settings.toml — the values

The idea

A viewer's title bar, buttons and panels are frosted glass: each samples the pixels directly behind it, blurs them, and lays a dark tint on top. The hard part is "directly behind it" — a viewer must see the backdrop and any viewers below it, but never itself or anything above (which would feed back).

The slot chain

The stage solves this with slots. A slot is a stage-sized, live composite of everything beneath one viewer. The stage assigns slot i to viewer i as its backdropSource:

backdrop = your app content (animated bg, gallery…) slot[0] = backdrop → viewer[0].backdropSource slot[1] = backdrop + capture(viewer[0]) → viewer[1].backdropSource slot[2] = backdrop + capture(viewer[0],1) → viewer[2].backdropSource … viewer[i] then samples its OWN rectangle out of slot[i].

Built in rebuildGlass(): viewers are sorted by z, the bottom one points straight at the backdrop leaf, and each higher viewer gets a slot composed of the backdrop plus the captureItem of every viewer below it:

function rebuildGlass() {
    …destroy old slots…
    let parts = topLayer.children sorted by z;
    for (let i = 0; i < parts.length; i++) {
        if (i === 0) parts[i].backdropSource = backdrop;          // bottom samples backdrop
        else       parts[i].backdropSource = makeSlot(parts.slice(0, i)); // composite below
    }
}
Flat slots, not slot-of-slot

Each slot composites leaf sources only — the backdrop and each lower viewer's captureItem (its glass-free contentLayer) — never another slot. A flat chain has no depth, so the front-most viewer can't "blank out" and there's no propagation lag. Capturing contentLayer (media + controls, minus that viewer's own glass) is what makes upper glass show lower viewers with their chrome — "compound" glass — without recursion.

It updates for free

Slots are built from live ShaderEffectSources, so viewer movement and animated content reflect instantly. Only structural changes — add / remove / re-order a viewer — trigger a rebuildGlass() (via topLayer.onChildrenChanged and selectViewer).

DsGlassBackground — the element

Any surface (a viewer's media region, a control set, a button) drops in a DsGlassBackground the same way: give it a context (the bundle the viewer hands its controls) plus its own size/corner radii. It samples its slice of the source slot, blurs, tints, masks corners, outlines:

DsGlassBackground {
    context: glassContext           // {source, viewerItem, refresh, enabled, tint, blur, …}
    topLeftRadius: 12; topRightRadius: 12
    fallbackColor: DsTheme.surface  // shown when glass is off / no source
}

Internally: a hidden ShaderEffectSource grabs source at sourceRect = the element's rect in stage coords (worked out with mapToItem + the refresh tick), a MultiEffect blurs it, a tint Rectangle sits on top, and a masked rounded shape + border finish it. When blurEnabled is false (or there's no source) it shows fallbackComponent or a solid fallbackColor instead.

MultiEffect.autoPaddingEnabled = false

MultiEffect defaults to padding/scaling its output for the blur — which silently offsets & scales the glass. Always set autoPaddingEnabled: false here. (And remember the glass origin uses mapToItem, which isn't reactive to re-parenting — hence the refresh tick the viewer bumps after layout; see Viewers.)

The glass tokens — how the blur is built

The blurred look isn't one value, it's a short pipeline. The grab of the backdrop is run through a MultiEffect, a tint is laid over it, the corners are masked round, and a thin stroke is drawn on top:

grab (ShaderEffectSource) → MultiEffect blur → tint Rectangle → rounded mask → border ▲ ▲ ▲ ▲ glassBlur + glassBlurMax glassTintOpacity glassRadius glassBorderWidth

The blur strength comes from two tokens working together, because that's how MultiEffect models blur:

  • glassBlur is the amount — a dial from 0.0 (no blur) to 1.0 (full blur).
  • glassBlurMax is the ceiling — the pixel radius that glassBlur = 1.0 actually reaches. Its meaningful range is 2 (subtle) to 64 (heavy); default 32. It decides what "full" means, and a bigger ceiling costs more (more texture samples per pixel).

So the on-screen radius is roughly the amount scaled by the ceiling:

radius ≈ glassBlur × glassBlurMax (px) │ │ amount 0–1 ─┘ └─ ceiling 2–64 (default 32) glassBlur 0.5 · glassBlurMax 32 → ~16 px medium — the default frosted look glassBlur 1.0 · glassBlurMax 32 → ~32 px heavy glassBlur 0.5 · glassBlurMax 64 → ~32 px same radius, finer quality, ~2× the cost

To make glass blurrier, turn the amount up first (it's free); only raise the ceiling when you want a radius bigger than the current max can reach. The remaining tokens are independent of the blur:

TokenRoleDefault
glassBlurBlur amount (0–1 dial).0.5
glassBlurMaxMax blur radius in px at amount 1.0 (2–64).32
glassTintOpacityOpacity of the dark surface tint over the blur (also baked into DsTheme.scrim).0.8
glassRadiusCorner radius of the rounded mask.12
glassBorderWidthWidth of the stroke outline (Tonal-10).1
Beyond the ceiling, and out-of-range amounts

glassBlur is clamped to 0.0–1.0, so a value above 1.0 just reads as full blur. If you need a wider radius than glassBlurMax 64 gives, MultiEffect also has blurMultiplier — it extends the radius without extra texture lookups (cheaper than a bigger blurMax) at some loss of quality. Waffles doesn't expose it as a token yet; add one the same way the others are wired if a design needs it.

The fullscreen scrim

Fullscreen reuses the same machinery. The scrim (which blocks the rest of the UI behind a fullscreen viewer) is blurred when the viewer's glass is on — it samples the fullscreen viewer's own backdropSource slot and blurs it full-screen — or a flat tint when glass is off. Resolved by _scrimModeFor() on the stage, overridable per viewer via fullscreenScrim.

The blur-mode scrim uses the same glassBlur / glassBlurMax by default, so it matches the viewer glass. If you want the backdrop behind a fullscreen viewer blurred differently from the viewer's own chrome, set scrimBlur / scrimBlurMax under [waffles.fullscreen] — the stage reads them into fullscreenScrimBlur / fullscreenScrimBlurMax, falling back to the glass tokens when unset.

Theming with DsTheme

DsTheme is a singleton (pragma Singleton, registered with QT_QML_SINGLETON_TYPE in CMake). It's a two-tier token system: raw primitives, then semantic roles that alias them. Components reference roles; re-skinning means re-pointing roles, not editing call sites.

QtObject {                       // DsTheme.qml
    // Tier 1 — primitives (the tonal ramp + brand hue)
    property color tonal0: "#191F25"; property color tonal10: "#2E3439"
    property color tonal30: "#5D6166"; property color white: "#FFFFFF"
    property color accent: "#00ADF7"
    // Tier 2 — semantic roles (alias primitives by purpose)
    property color surface: tonal0; property color surfaceVariant: tonal10
    property color stroke: tonal10; property color surfaceText: white
    property color accentText: white; property color track: tonal30
    readonly property color scrim: Qt.rgba(tonal0.r, tonal0.g, tonal0.b, glassTintOpacity)
    // Glass tokens (non-colour; palette can't carry these)
    property real glassTintOpacity: 0.8; property real glassBlur: 0.5
    property int  glassBlurMax: 32; property real glassRadius: 12; property real glassBorderWidth: 1
}

The stage assigns Qt's palette from these roles once, high in the tree, so Qt Quick Controls inherit it (dark surfaces, white text, Tonal-10 strokes, accent highlight):

palette.window: DsTheme.surface;   palette.text: DsTheme.surfaceText
palette.highlight: DsTheme.accent; palette.highlightedText: DsTheme.accentText  // …etc
Never name a role on<Capital>

QML reserves identifiers starting with on+uppercase for signal handlers, so a property color onSurface silently fails to resolve (reads as black). That's why the roles are surfaceText / accentText, not the Material-style onSurface / onAccent.

See Docs/articles/color_system.md for the full rationale (primitives vs roles, mapping to Qt's palette, what palette deliberately doesn't carry).

Settings

Waffles values live in Examples/AssetViewer/settings/waffles_settings.toml, all under a [waffles] table, and are merged into the app_settings collection by listing the file in engine.toml:

# engine.toml
[engine.extra]
app_settings = [ "content_settings.toml", "menu.toml", "waffles_settings.toml" ]

# waffles_settings.toml
[waffles]
controlsIdleSeconds = 4
[waffles.theme]               # seeds DsTheme
tonal0 = "#191F25"
accent = "#00ADF7"
glassTintOpacity = 0.8
[waffles.viewer]
minWidth = 400
imageWidth = 1400
[waffles.fullscreen]
margin = 0
scrimOpacity = 0.6
#scrimBlur = 0.7        # optional; defaults to glassBlur
#scrimBlurMax = 48      # optional; defaults to glassBlurMax

QML reads them through a DsSettingsProxy with the waffles prefix, so getFloat("viewer.minWidth") resolves waffles.viewer.minWidth:

DsSettingsProxy { id: appSettings; target: "app_settings"; prefix: "waffles" }
Colours: token vs setting

DsTheme seeds its Tier-1 primitives & glass tokens from [waffles.theme] (via its own DsSettingsProxy { prefix: "waffles.theme" }), falling back to built-in defaults; the semantic roles are still derived in DsTheme. So white-label colours live in the TOML, while DsTheme remains the runtime source of truth (and is still writable for runtime re-skinning). See settings_guide.md for the full settings format.
