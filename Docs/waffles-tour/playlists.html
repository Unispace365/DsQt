7 · Playlists & transitions

A playlist plays an ordered list of templated slides full-stage, in the stage's presentation layer (below the viewer layer). Two modes: ambient auto-advances on a timer (the attract loop); interactive waits for next/prev and shows a presentation controller. Slides change with a transition — a built-in, a per-slide choice, or a custom one you write in QML/JS or as an RHI shader.

Key files

  • waffles/playlist/DsPlaylistViewer.qml — the full-stage slide player (dual-slot, transitions, auto-advance)
  • waffles/playlist/DsMediaTemplate.qml — the default full-screen media slide template
  • waffles/playlist/PresentationController.qml — movable/collapsible prev·play·count·next·close bar (interactive only)
  • waffles/DsWaffleStage.qml — openPlaylist() / closePlaylist() + the playlist settings
  • Bridge/qml/DsContentLauncherModel.qml — slidesFor(record) turns a playlist record into slide descriptors
Ambient vs interactive

Ambient (ambient: true) auto-advances on a per-slide hold time and shows no controller — it's hands-off attract content. Interactive (ambient: false) holds each slide until next/prev (keyboard ← → / PageUp · PageDown, or the controller) and shows the PresentationController. Both loop (wrap past the ends).

Opening & closing

The stage owns playlist lifetime. openPlaylist() creates a viewer in the presentation layer (and, for interactive, a controller in modal1); a second call replaces the first with a crossfade. closePlaylist() fades the current one out.

// props: slides (required), ambient, and optional per-playlist transition overrides.
let pv = stage.openPlaylist({
    "slides":  launcherModel.slidesFor(playlistRecord),  // ordered descriptors
    "ambient": true,
    "defaultTransition":  "fade",   // optional — else [waffles.playlist] setting
    "transitionDuration": 500     // optional — else the setting
});
// …later
stage.closePlaylist();

A slide descriptor (from DsContentLauncherModel.slidesFor) is a plain object:

{ uid, title, kind, typeUid, media, thumbnail,
  holdTime,           // seconds: slide → platform → defaultHoldTimeSeconds
  disableTransition,  // bool — skip this slide's transition (cut)
  transition,         // string — per-slide transition name (see below)
  record }            // the raw bridge record, for template-specific fields

Templates — which QML renders a slide

Each slide is rendered by a template chosen from the slide's typeUid. Register your template components on the stage; anything unmapped falls back to DsMediaTemplate (full-screen image/video, fill or fit, autoplay/loop):

DsWaffleStage {
    id: stage
    playlistTemplateByTypeUid: ({
        "ykAdRYGJeuXI": myMediaTemplateComp,   // typeUid → Component
        "ScCWZMV8UNbx": particleTextComp
    })
}

A template receives the slide as a slide property and an active bool (true while it is the visible/incoming slot — gate video playback on it):

Item {                       // a slide template
    property var  slide          // the descriptor above
    property bool active         // is this slot live? play video only when true
    // read slide.media / slide.record.<field> to render
}

The presentation controller

Interactive playlists get a PresentationController — a movable, collapsible bar (prev · play/pause · i / N · next · close), mirroring the fullscreen controller's tab+bar UX. Ambient playlists get none. Play/pause toggles the viewer's paused flag; close fires closeRequested → closePlaylist().

Transitions

The viewer renders with two stacked slots: _front shows the current slide; on advance the incoming slide loads into _back, the chosen transition animates, then _back is promoted to front. Built-ins:

nameeffect
noneinstant cut (no animation)
fadeincoming sits under at full opacity, outgoing fades out on top — a dissolve with no background flash, even when a slide is letterboxed
slideLeft / slideRightincoming slides in from the right / left; the two slots stay edge-to-edge so the frame is always covered
Why fade doesn't flash the background

A naïve crossfade (old 1→0 while new 0→1) leaves both slots semi-transparent mid-way, so the opaque stage background shows through — worst over a fit slide's letterbox bars. Instead the incoming slide is drawn fully opaque underneath and the outgoing fades out on top: the visible surface is always backed by a 100%-opaque slide.

Choosing a transition

Per advance, the name is resolved in this order (first match wins):

#source
1slide.disableTransition is true → none
2slide.transition (per-slide name)
3the playlist's defaultTransition (openPlaylist prop)
4[waffles.playlist] defaultTransition setting
5hard fallback "fade"
Per-slide names need a schema field

The demo schema only carries disable_transition (a checkbox), so today per-slide "none" is authorable but per-slide names resolve to empty (→ the playlist default) until a transition field is added to the slide type. The viewer reads it via DsContentLauncherModel.transitionField (default appKey "transition").

Settings

[waffles.playlist]
defaultTransition  = "fade"   # none | fade | slideLeft | slideRight | <custom name>
transitionDuration = 500      # per-slide transition, ms
crossfadeMs        = 400      # whole-playlist crossfade (ambient ↔ interactive), ms

Custom transitions

Anything that isn't a built-in name is looked up in the viewer's customTransitions map (name → function). A custom transition is a plain JS function that animates the two slot Items and signals completion:

// (front, back, durationMs, done)
//   front    – the OUTGOING slot Item (x / y / opacity / z / scale / rotation …)
//   back     – the INCOMING slot Item; the new slide is already loaded into it
//   durationMs – the resolved transitionDuration
//   done()   – CALL EXACTLY ONCE when finished → promotes back→front, schedules the next advance
// Before calling you the viewer sets back.z = 1, front.z = 0 (incoming on top).

A simple "wipe up" using a one-off NumberAnimation:

stage.playlistCustomTransitions["wipeUp"] = function(front, back, durationMs, done) {
    back.y = back.height;                 // start the incoming below the frame
    back.opacity = 1;
    const anim = wipeComp.createObject(back, {
        "target": back, "property": "y", "to": 0, "duration": durationMs
    });
    anim.finished.connect(function() { anim.destroy(); done(); });
    anim.start();
};
// where, defined once in your app:
//   Component { id: wipeComp; NumberAnimation { easing.type: Easing.InOutCubic } }

Select it like any other name — set the playlist's defaultTransition (or a slide's transition) to "wipeUp".

Registering custom transitions

Register once on the stage — stage.playlistCustomTransitions (a name → function map) is threaded into every playlist openPlaylist() creates, so a custom name in any slide's or playlist's transition resolves to it. You can set the whole map at startup, or add entries before opening. A single openPlaylist({ "customTransitions": … }) call overrides it per-playlist, and the same map also lives on the viewer as DsPlaylistViewer.customTransitions if you hold an instance directly.

DsWaffleStage {
    id: stage
    playlistCustomTransitions: ({
        "wipeUp":   wipeUpFn,     // name → function(front, back, durationMs, done)
        "dissolve": dissolveFn
    })
}

The GPU / RHI path

The hook hands you live QML Items, so anything QML can animate works — including GPU shader transitions. Qt's ShaderEffect compiles to the active RHI backend (D3D11/Vulkan/Metal/GL) from a precompiled .qsb, so a shader dissolve/wipe runs on the GPU with no C++. Wrap each slot as a live texture and blend with an animated uniform:

// fxComp — define once in your app (front/back are the slots; emits finished())
Item {
    id: fx; anchors.fill: parent; z: 10
    property Item from; property Item to; property real progress: 0
    signal finished()
    ShaderEffectSource { id: fromTex; sourceItem: fx.from; live: true; hideSource: true }
    ShaderEffectSource { id: toTex;   sourceItem: fx.to;   live: true; hideSource: true }
    ShaderEffect {
        anchors.fill: parent
        property variant source:  fromTex
        property variant dest:    toTex
        property real    progress: fx.progress
        fragmentShader: "qrc:/shaders/dissolve.frag.qsb"
    }
    NumberAnimation on progress { from: 0; to: 1; running: true; onFinished: fx.finished() }
}
stage.playlistCustomTransitions["dissolve"] = function(front, back, durationMs, done) {
    const fx = fxComp.createObject(back.parent, { "from": front, "to": back });
    fx.finished.connect(function() { fx.destroy(); done(); });
};
Compile the shader to .qsb

ShaderEffect.fragmentShader wants a .qsb (the RHI-portable form), produced from a .frag by the Qt Shader Tools — qt6_add_shaders(target "x" FILES dissolve.frag) in CMake, or qsb on the command line. The shader gets source/dest as sampler2Ds, progress in the uniform block, plus the standard qt_TexCoord0 / qt_Matrix / qt_Opacity.

For full control there is also the C++ scene-graph route — a QQuickItem with a custom QSGNode/QSGMaterial, or a QSGRenderNode — driven by the same hook. There's no scaffolding for it in Waffles today; reach for it only if a ShaderEffect can't express the effect, since the shader path covers virtually all dissolve/wipe/distortion transitions.

The whole-playlist crossfade

Distinct from per-slide transitions: when you switch playlists (ambient ↔ interactive) or close one, the entire viewer fades. openPlaylist keeps the outgoing viewer fully opaque underneath while the new one fades in on top (opacity 0→1), then destroys the old — again, no background flash. The outgoing controller fades out with it. Duration is [waffles.playlist] crossfadeMs.
